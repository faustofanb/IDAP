# 项目宪章：进销存多租户后台 + 智能问数平台

## 第一条——规范至上

- 规格（spec）、实施计划（plan）与任务清单（tasks）是唯一事实来源（Source of Truth）。
- 任何实现必须可溯源至已批准的规格条款与验收标准。

## 第二条——以用户结果为中心

- 明确定义受众（分析师、运营、管理员）及其可度量价值（KPI、时延、准确率等）。
- 模糊需求以 `[NEEDS CLARIFICATION: ...]` 标注，未澄清前不得进入开发。

## 第三条——测试驱动与契约优先

- 先契约（JSON Schema / 类型）与测试，再实现；未具备单元/集成/契约测试覆盖的任务不得关闭。
- 跨服务通信以结构化 JSON 契约为准，契约存于 `specs/<feature>/contracts/` 并生成三端类型。

## 第四条——架构护栏

- 前端：Vue 3 + TypeScript + Vite + Pinia + Element Plus + ECharts；WebSocket 主通道，REST 兜底。
- 后端：Java 21 + Spring Boot + JPA + WebSocket + Redis + Kafka（MVP 可内存桩件）；CQRS；JWT 认证。
- AI 服务：Python + FastAPI + LangChain + LangGraph；不得直接执行 SQL，仅输出结构化请求。
- 通信：结构化 JSON（WS/HTTP），附带 `correlationId` 与遥测。

## 第五条——CQRS 与事件

- 读写分离（Command/Query）；事件驱动集成记录生产者、消费者、模式与投递保证。

## 第六条——安全与合规

- 身份认证采用 JWT；鉴权基于 RBAC，最小授权单元为 URL/资源；菜单仅作导航。
- 多租户：所有请求必须显式携带 `tenantId`；跨租户默认禁止，运营跨租需审批并全量审计。

## 第七条——性能期望

- 为 API、WebSocket 与 AI 往返声明 p95 延迟预算与吞吐目标；计划中记录并发与容量策略。

## 第八条——可观测性

- 标准化日志、追踪与指标；端到端关联 `correlationId`；提供运行健康与错误预算视图。

## 第九条——演进流程

- 对本宪章与规格的变更需干系人共识，并包含影响评估与迁移计划。
- 当前阶段仅维护“特性级”的实施计划与任务清单：`specs/<feature>/{plan.md,tasks.md}`（本项目为 `specs/001-intelligent-querying/`）。
- 更新义务：
  - 每次任务完成后，必须同步更新：
    - `specs/<feature>/plan.md`（里程碑/范围与风险状态）
    - `specs/<feature>/tasks.md`（完成勾选/遗留与后续动作）
  - 提交信息与文档“更新记录”需体现日期、作者与要点，PR 描述建议引用相应条目。
- 当项目扩展为“多特性”时，可增设 `specs/000-project/{plan.md,tasks.md}` 作为项目级总览与横向事项归档。

## 第十条——文档规约

- 根目录 `docs/` 仅存放架构综述与规约类文档；实现细节/运维/开发手册等放各子项目 `*/docs/`。
- 新增文档（除本宪章外）须使用数字前缀排序（如 `01-架构综述.md`、`02-业务域与多租户.md`）。
- 所有文档统一使用简体中文；必要术语可中文后括注英文原词。

## 第十一条——业务域与模块边界

- 业务定位：多租户“进销存”后台（采购-销售-库存）+ 报表 + 权限治理。
- 模块清单：租户、机构、用户、菜单&URL、权限（RBAC/数据权限）、报表、采购、销售、库存（入/出/调拨/盘点/退货等）。
- 多租户隔离：逻辑隔离（tenant_id + 全局过滤）优先；跨租默认禁止；运营跨租需特权审批并审计。

## 第十二条——AI 范围与约束

- 范围：仅“智能数据查询 + 报表分析”。
- 约束：AI 不得直连数据库、不直接执行 SQL；仅生成 `RequestEnvelope`（BusinessRequest/SQLRequest）供受控引擎执行；响应需附带遥测与来源说明。
